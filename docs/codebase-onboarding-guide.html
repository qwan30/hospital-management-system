Tổng quan hệ thống & Sơ đồ phụ thuộc Module

Hệ thống Quản lý Bệnh viện (HMS) được cấu trúc dưới dạng một mô hình Monolith phân tầng sạch sẽ. Giao diện frontend Next.js 16 giao tiếp trực tiếp với Spring Boot 3.3 backend và cơ sở dữ liệu PostgreSQL.

Luồng Phụ Thuộc Module Backend

Hệ thống monolith mô-đun ở backend áp dụng nghiêm ngặt các ranh giới kiến trúc sạch (clean architecture):
start (chạy ứng dụng, phụ thuộc vào controller, application, infrastructure) ➔ controller (phơi bày REST API, phụ thuộc vào application) ➔ application (xử lý logic nghiệp vụ, phụ thuộc vào domain và sử dụng repositories) ➔ infrastructure (truy cập cơ sở dữ liệu và cấu hình JPA/Flyway) ➔ domain (chứa thực thể cốt lõi độc lập hoàn toàn).
Tầng thành phần Công nghệ & Tệp tin Mô tả & Trách nhiệm chính
Frontend Client Next.js 16, React 19, TS
web/		 Ứng dụng client động sử dụng React Server & Client Components, lưu trữ trạng thái qua sessionStorage, quản lý định tuyến RBAC.
API Client Wrapper Native Fetch, TS Interfaces
web/src/lib/		 Tập hợp helper thực hiện gọi API được phân tách theo nghiệp vụ. Wrapper chính trong api-client.ts tự động đính kèm token Bearer JWT.
REST Controllers REST Controllers Spring Boot
backend/controller/		 Expose các endpoints dưới dạng `/api/v1/*`. Thực hiện xác thực payload đầu vào, kiểm tra quyền truy cập và bọc phản hồi trong định dạng envelope `ApiResponse`.
Tầng dịch vụ (Application Services) Spring Services & DTOs
backend/application/		 Điều phối các quy trình nghiệp vụ, kiểm tra tính toàn vẹn dữ liệu, và quản lý các giao dịch cơ sở dữ liệu thông qua annotation `@Transactional`.
Thực thể cốt lõi (Domain Entities) JPA & Hibernate entities
backend/domain/		 Các thực thể biểu diễn dữ liệu cốt lõi (như `AppointmentEntity`, `PatientEntity`). Độc lập hoàn toàn với framework và thư viện ngoài.
Tầng hạ tầng (Infrastructure) Spring Data JPA, Flyway, Postgres
backend/infrastructure/		 Thực thi các câu lệnh truy vấn database, thiết lập bộ giải mã token JWT, và chứa các tệp di cư schema Flyway SQL.

Chiến lược kiểm thử toàn diện

Dự án áp dụng quy chuẩn kiểm thử nghiêm ngặt, yêu cầu độ bao phủ tối thiểu 80% đối với mã nguồn mới/thay đổi. Kiểm thử được phân bổ rõ ràng giữa Backend (JUnit 5, Mockito) và Frontend (Vitest, RTL, Playwright).

1. Khóa ghi cơ sở dữ liệu chống trùng lặp (Pessimistic Write Locking)

HMS ngăn chặn tình trạng đặt lịch trùng lặp phòng khám hoặc khung giờ bác sĩ bằng cách áp dụng khóa độc quyền ở tầng dữ liệu.
Mô tả mã nguồn
Sử dụng Pessimistic Write Lock khi truy vấn khung giờ rảnh
File liên quanTimeSlotRepository.java
Chi tiết logicChú thích @Lock(LockModeType.PESSIMISTIC_WRITE) trên phương thức lockWindow() và findByIdForUpdate(). Khi có nhiều luồng tranh chấp đăng ký cùng một khung giờ, DB sẽ khóa dòng dữ liệu đó, buộc các luồng đăng ký sau phải chờ cho đến khi giao dịch hiện tại hoàn tất (Commit hoặc Rollback), giúp loại bỏ hoàn toàn rủi ro Overbooking.

2. Ghi nhật ký vi phạm phân quyền ở tầng bộ lọc (Filter Level Denial Audit)

Ghi nhận toàn bộ sự cố truy cập trái phép trực tiếp tại ranh giới servlet filter trước khi yêu cầu đi vào controller Spring.
Mô tả mã nguồn
Bộ lọc ghi nhật ký truy cập trái phép
File liên quanAuthorizationDenialAuditFilter.java
Chi tiết logicFilter bắt gói phản hồi có HTTP status 401 hoặc 403. Sau đó, trích xuất mã số định danh tác nhân (actorId), phương thức, địa chỉ URL đã được làm sạch cấu trúc, định dạng và ghi trực tiếp bản ghi kiểm toán bảo mật SECURITY_ACCESS_DENIED vào bảng CSDL audit_logs.

3. Tích hợp trực tiếp API REST của Gmail với cơ chế tự động xoay Token

HMS xây dựng bộ gửi email nhẹ thông qua kết nối HTTP thô đến Google Cloud API, giảm sự phụ thuộc vào các SDK lớn.
Mô tả mã nguồn
Giao thức trao đổi JWT và xoay vòng Refresh Token
File liên quanGmailApiClient.java
Chi tiết logicMỗi khi gửi mail (Xác nhận đặt lịch, Báo cáo kết quả khám kèm PDF đơn thuốc), client gửi request POST trao đổi OAuth2 Refresh Token lấy Bearer Access Token dùng một lần, sau đó đóng gói thư điện tử chuẩn MIME thô dưới dạng URL-Safe Base64 và chuyển tiếp qua API Google.

4. Nạp Font UTF-8 hệ thống để hiển thị đúng Tiếng Việt có dấu trong PDFBox

Xử lý hiển thị ký tự Unicode tiếng Việt trong đơn thuốc xuất bản dạng PDF mà không làm vỡ hoặc lỗi font chữ.
Mô tả mã nguồn
Duyệt qua danh sách Font khả dụng của hệ điều hành
File liên quanPrescriptionPdfService.java
Chi tiết logicDuyệt tìm font TrueType khả dụng (Arial.ttf trên Windows, DejaVuSans.ttf trên Linux) và nạp qua PDType0Font.load() của Apache PDFBox. Nếu không tìm thấy, hệ thống sẽ fallback về font chuẩn Helvetica của PDF. Điều này đảm bảo đơn thuốc hiển thị đúng tên tiếng Việt có dấu.

5. Cấu hình Demo Seed dữ liệu tải lớn (Performance Simulation Seeding)

HMS xây dựng một bộ khởi tạo dữ liệu mẫu cực lớn giúp lập trình viên kiểm thử hiệu năng tìm kiếm và phân trang trực tiếp.
Mô tả mã nguồn
Seeding hàng trăm bản ghi mẫu thông qua cấu hình Spring Properties
File liên quanSeedDataService.java
Chi tiết logicNhận thuộc tính từ lớp NonBillingDemoSeedProperties. Nếu được kích hoạt (enabled = true), ứng dụng tự động chạy CommandLineRunner chèn hàng trăm phòng ban, bác sĩ, bệnh nhân, lịch hẹn mẫu và hàng ngàn log kiểm toán, giúp mô phỏng chính xác môi trường sản xuất.

6. Mặt nạ bảo vệ thông tin định danh cá nhân bệnh nhân (CCCD UI Masking)

Tuân thủ nguyên tắc thiết kế bảo mật riêng tư (Privacy-by-Design) bằng cách che giấu các thông tin nhạy cảm trên màn hình hàng đợi công cộng.
Mô tả mã nguồn
Che ký tự căn cước công dân trên bảng điều phối của điều dưỡng
File liên quanweb/src/lib/staff-queue.ts
Chi tiết logicHàm maskIdentifier(value) nhận chuỗi CCCD gốc, thực hiện cắt chuỗi và chèn ký tự sao (ví dụ: hiển thị 079***012). Điều này giúp bệnh nhân và điều dưỡng quan sát mã số định vị hàng đợi mà không lộ số căn cước công dân thực tế trên màn hình hiển thị.

7. Thuật toán đo lường tải khám và điều phối Bác sĩ tại phòng tiếp nhận

HMS tính toán số lượng bệnh nhân tại từng trạng thái chờ khám của bác sĩ để tối ưu hóa thời gian tiếp đón bệnh nhân.
Mô tả mã nguồn
Tổng hợp số lượng bệnh nhân chờ đợi thực tế theo bác sĩ
File liên quanweb/src/lib/staff-queue.ts

8. Phân mức độ ưu tiên hàng đợi qua cảnh báo màu thời gian chờ (SLA Alerts)

Chỉ báo trực quan cho điều dưỡng biết bệnh nhân nào đã chờ quá thời gian cam kết dịch vụ (SLA) tại phòng khám.
Mô tả mã nguồn
Mã hóa màu dựa trên khoảng thời gian tính bằng phút
File liên quanweb/src/lib/staff-queue.ts
Chi tiết logicHàm getWaitBadgeClass(waitMinutes) sẽ kiểm tra: nếu thời gian chờ lớn hơn hoặc bằng 40 phút, trả về màu đỏ báo động (danger); từ 20 phút trở lên trả về màu vàng cảnh báo (warning); dưới 20 phút là màu xanh an toàn (success).

9. Sổ cái kho dược phẩm cập nhật theo biến động giao dịch (Inventory Ledger Movements)

HMS bảo vệ tính toàn vẹn tồn kho bằng cách cấm giảm trực tiếp số lượng thuốc trên kệ, thay vào đó bắt buộc thông qua giao dịch biến động.
Mô tả mã nguồn
Tính toán delta tồn kho và lưu vết biến động (Movements)
File liên quanInventoryWriteService.java
Chi tiết logicHàm recordMovement() nhận delta biến động âm hoặc dương. Nó kiểm tra nếu tồn kho sau biến động bị âm sẽ ném ra lỗi xung đột (ConflictException). Nếu hợp lệ, hệ thống tạo bản ghi InventoryMovementEntity, lưu trữ vết loại biến động (RESTOCK, DISPENSE) và cập nhật số lượng tồn thực tế.

10. Scheduler múi giờ nội địa & Thiết kế nạp Clock giả lập trong kiểm thử

Thiết lập chạy tiến trình tự động gửi email nhắc lịch tái khám đúng múi giờ Việt Nam và cho phép giả lập thời gian khi chạy Test.
Mô tả mã nguồn
Tích hợp Scheduled Cron và tiêm Bean Clock để mock thời gian
File liên quanReminderService.java
Chi tiết logicChú thích @Scheduled(cron = ..., zone = "Asia/Saigon") chạy quét mỗi 15 phút. Thay vì dùng Instant.now() tĩnh, dịch vụ tiêm vào đối tượng Clock. Khi chạy unit test, ta có thể dễ dàng chế tạo Clock giả lập thời gian trôi hoặc ngày bất kỳ để kiểm tra xem tiến trình gửi mail nhắc nhở có hoạt động chuẩn hay không.
