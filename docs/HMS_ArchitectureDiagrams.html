Hospital Management System Architecture

Repository-aligned diagrams for the current backend-first system: Spring Boot 3.3 multi-module backend, PostgreSQL with pgvector, Flyway migrations, optional Gemini and Gmail integrations, and a frontend that remains a scaffold.

1. Runtime System Context

Current executable surface: browser clients call the Spring Boot API on port 8080. Docker Compose starts PostgreSQL and the backend; the frontend container remains commented out until the UI is implemented.

2. Backend Module Boundaries

The Maven reactor is DDD-oriented. The start module is the composition root; lower modules are protected by module-boundary tests and should not import higher layers.

3. Data and Integration Architecture

PostgreSQL is the persistence contract. Flyway migrations create and evolve the schema; integrations are optional and degrade without blocking core workflows unless their specific operation requires them.

4. Role Workflow Map

This view maps user roles to implemented backend capability groups. It intentionally avoids frontend-only promises that the current repository does not support.

Current Product Boundary

  • • Backend API is the implemented production surface.
  • • Frontend is planned but not production-ready in this repo.
  • • Docker Compose runs PostgreSQL and backend only.

Backend Shape

  • • Java 17, Spring Boot 3.3, Maven reactor.
  • • Modules: domain, infrastructure, application, controller, start.
  • • ModuleBoundaryTest guards compile-time direction.

Persistence Contract

  • • PostgreSQL 15 with pgvector image.
  • • Flyway V1-V10 manages schema changes.
  • • JPA validates schema at startup.
